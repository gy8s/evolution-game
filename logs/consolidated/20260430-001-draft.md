## Bugs and Issues

### BUG-001: ERR_ALIVE_WITH_ZERO_FITNESS — state integrity failure
- **Severity**: ERROR / P0
- **Code**: ERR_ALIVE_WITH_ZERO_FITNESS
- **Occurrences**: 2 in 20260430-162018 (run 2 turn 68, run 9 turn 64). Both have ant trail as encounterAtDeath, deathContext=null, dead=false.
- **Description**: Player reaches fitness=0 but `die()` is not called. In run 2, hydration is also 0 — the dehydration damage tick in `startTurn()` should be triggering `die()`, but deathContext is null, suggesting the death path was not completed. This likely means poison or dehydration damage in `startTurn()` reduced fitness to 0 but the subsequent `if (player.fitness <= 0) { die(...) }` check did not fire, or fired but failed to set deathContext before the snapshot was taken.
- **This is a game engine bug, not a bot bug.** Any player (human or bot) on zero fitness without the death being recorded is a broken game state.
- **Status**: OPEN — must fix before results are reliable

### BUG-002: Run summary duplication — each death recorded twice
- **Severity**: WARNING / reporting quality
- **Description**: Every death produces two run summary entries: "player died" and "player died before next action". This halves the apparent run count and makes per-run analysis harder. Present in all bot runs.
- **Root cause**: `summariseCurrentBotRun("player died")` is called at the end of a step when death is detected, then called again at the start of the next step with "player died before next action". The fix is to remove the end-of-step call and let the start-of-next-step call handle it (stopRandomBot already handles the turn-limit edge case correctly).
- **Status**: OPEN — fix is straightforward

## Code Issues (from direct code review)

### CODE-001: Missing `}` in `chooseIntelligentBotAction` — dead code block
- **Severity**: Low / code hygiene
- **Location**: `chooseIntelligentBotAction`, the block starting `if (enc && (isDangerous || isThreatClass) && preferredEscapeActions.length)`
- **Description**: The outer `if` block is never closed with its own `}`. The inner duplicate `if` block is unreachable dead code. Runtime behaviour is unaffected (the `return` fires when the condition is true), but the code is confusing and should be cleaned up.

### CODE-002: `intelligentMoveTowardWater` targets water tiles, not land-adjacent-to-water
- **Severity**: Medium — causes drowning risk
- **Description**: The function finds the nearest water tile and moves toward it. This can result in the bot stepping onto a water tile (and drowning, or being taken by a water predator). The correct target is a land tile adjacent to water — a position from which `canDrinkAtWaterEdge()` returns true without entering water. This fix was prepared on branch `claude/intelligent-bot-agent-oCneM` but was not carried to main.

### CODE-003: `preferredEscapeActions` is wrong for arboreal/grapple threats (constrictor)
- **Severity**: High — directly causes ~4 deaths per 1000 actions
- **Description**: `preferredEscapeActions = climbAction ? [climbAction, ...moveActions] : moveActions` includes climbing as the first option for ALL dangerous encounters. This is correct for ground predators (Gastornis, mesonychid) but wrong for arboreal constrictors: climbing sends the bot into the constrictor's layer. The grapple/ambush/arboreal flee response should descend or move laterally, not climb. This is likely the single biggest fixable bot behaviour issue right now.

## Observations

### OBS-001: Constrictor is the highest-value fix target
The large constrictor (`dangerProfile: "grapple"`, `temperament: "ambush"`) kills 4 of 17 durated deaths in the latest run, including a 4-turn run (run 7). It lives in upper layers. The bot currently climbs into it. Inverting the escape direction for grapple/ambush encounters is a small code change with a large expected impact.

### OBS-002: Gastornis kills are a hydration trap
Runs 11 and 12 are long (79 and 101 turns) but both die to Gastornis on the ground floor. Both have moderate-to-zero hydration at death. The most likely explanation: the bot descends to ground level to reach water, and Gastornis is waiting there. The bot needs to treat water access as a danger zone when ground predators have recently been active, not just react to the encounter once it appears.

### OBS-003: Ant trail and termite mound are risky food the bot still eats
Ant trail has mild poison and is treated as forage — `intelligentFindSafeEat` will try it if the bot hasn't eaten it before (unknown severity). Termite mound (run 6) has a guardian mechanic that spawned a termite swarm and delivered severe/lethal poison. Both should be on a blacklist for `intelligentFindSafeEat` unless the bot is in critical-energy desperation and has no alternative.

### OBS-004: Parasite deaths (2) suggest groom trigger needs lowering
Runs 13 and 15 die from parasites (load 2 and 4). The current trigger grooms at `parasites >= 2`. Run 13 dies with load 2, meaning the groom either didn't fire or didn't reduce load enough before the death tick. The threshold may need to be `>= 1` (groom as soon as any parasites are detected) or the groom action may need to be repeated.

### OBS-005: WARN_STATIC_QUEUED_ENCOUNTER_DISAPPEARED — recurring
One occurrence in 162018 (run 15, step 714). Same pattern as earlier runs — investigation of one encounter causes a queued static encounter to vanish. Ongoing, low priority.

### OBS-006: Death cause aggregation still imprecise
Run 2 dehydration death is aggregated as "ant trail" because ant trail was the `encounterAtDeath`. Run 9 fitness-collapse is also attributed to ant trail. Run 15 parasite death is attributed to "ant swarm" (the visible encounter, not the actual cause). The aggregation should prefer `context.source` values like "dehydration", "parasites", "critical-poison-timer" over `encounterAtDeath.name` when those sources are specific and meaningful.

## Root Cause of Survival Ceiling

### Is the bot reactive or state-based?

The bot is **purely reactive and stateless**. `chooseIntelligentBotAction` is a function that reads current game state, evaluates a priority-ordered rule list, and returns one action. The next call has no memory of the previous call. Every turn starts from zero.

### Is that sufficient?

No. It is the root cause of the survival ceiling.

A stateless reactive function cannot:
- Remember where a predator was 5 turns ago, so it walks back into the same territory
- Plan a route to water that avoids a danger zone it has already seen
- Persist a goal across turns — if it was moving toward water and an encounter interrupts, the water goal is lost
- Distinguish "currently safe" from "have been safe for 20 turns because I am well-positioned" — it cannot preserve a good position deliberately
- Detect that it is oscillating between two tiles and correct course

The practical consequence is that every rule added to the function addresses one specific observable symptom (constrictor kills, ant trail deaths, water tile entry) but the underlying failure mode remains: the bot has no world model, no memory, and no goal. A new set of symptoms will surface after every patch cycle. This is why the heuristic approach has been hitting a new ceiling with each patch series.

### What is structurally wrong with the current design?

The current design conflates two things that should be separate:

1. **What do I know?** (memory, goal, spatial history)
2. **What should I do right now?** (action selection given what I know)

The bot only has the second question. It has no answer to the first because it has no persistent cross-turn state beyond what the game engine already exposes. It knows fitness, hydration, energy, and the current encounter — all of which are point-in-time snapshots. It knows nothing about what happened before this turn.

### What is the structural fix?

Two components, not one:

**Component 1: Cross-turn bot memory in `botState`**
A persistent record attached to `botState` that survives across turns within a run:
- `recentThreats`: list of `{name, x, y, turn}` for predators/hazards seen in the last N turns
- `currentGoal`: `{type, target, priority}` — a goal that persists until completed or explicitly abandoned
- `dangerZones`: rough spatial record of areas that have produced threats

Without this, a state machine is still reactive — it just reorganises the same rule list into named modes, each computed fresh from current game state. That is better-structured, but it does not change the fundamental class of bot.

**Component 2: Goal-directed action selection**
With persistent state, `chooseIntelligentBotAction` becomes: "given my current goal and memory, what is the best action this turn?" rather than "given what I can see right now, which rule fires first?"

The state machine (P4) is the right next step **only if** it reads from persistent cross-turn state, not just from current turn snapshots. If mode is recomputed from scratch each turn from current game state alone, it is architecturally equivalent to the existing priority list — better organised, but still reactive.

### Honest audit of the P0–P9 list

| Item | Type | Justification |
|------|------|---------------|
| P0 — ERR_ALIVE_WITH_ZERO_FITNESS | Engine bug fix | Not a heuristic. Must fix regardless. |
| P1 — Constrictor escape direction | Heuristic fix | Corrects a wrong rule. Will help but does not change the architecture. |
| P2 — Water navigation fix | Heuristic fix | Corrects target selection. Will help but does not change the architecture. |
| P3 — Forage blacklist | Heuristic addition | More rules. Will help in the short term. Will hit a new ceiling. |
| P4 — Survival mode state machine | **Structural — conditionally** | Only structural if modes read from persistent cross-turn state. Otherwise it is a reorganised rule list. |
| P5 — Run summary duplication | Reporting fix | Not a heuristic. |
| P6 — Parasite groom threshold | Heuristic tweak | More rules. Same class as P1–P3. |
| P7 — Death cause aggregation | Reporting fix | Not a heuristic. |
| P8 — Terminal trace | Observability | Not a heuristic. |
| P9 — Code cleanup | Hygiene | Not a heuristic. |

**Conclusion**: P1, P2, P3, P6 are more heuristics. They should be implemented alongside, not instead of, the structural change — they will reduce deaths in the short term and make the structural change easier to evaluate. But if P4 is implemented without cross-turn memory, the next plateau will appear quickly.

The proposal is: implement P0/P5/P7/P8/P9 as low-risk fixes, implement P1/P2/P3/P6 as targeted heuristic improvements, and treat P4 as the structural investment — but define it to include a persistent `botState.memory` object from the start so it is not just a reorganised rule list.

## Notes for Next Patch (priority order)

**P0 — State integrity: ERR_ALIVE_WITH_ZERO_FITNESS**
Investigate why fitness reaches 0 without `die()` being called. Fix the death path so deathContext is always written before the snapshot is captured. This is a game engine bug. Patch gate should be red until resolved.

**P1 — Constrictor escape direction (CODE-003)**
Change `preferredEscapeActions` to be threat-type aware. For `grapple`/arboreal encounters, use `descendAction` or lateral moves, not climb. Expected to cut constrictor deaths significantly.

**P2 — Water navigation fix (CODE-002)**
`intelligentMoveTowardWater` should target land-adjacent-to-water tiles, not water tiles. Eliminates drowning-via-navigation and reduces water-predator exposure at approach.

**P3 — Forage risk blacklist**
Exclude ant trail, termite mound, ant swarm (and any encounter with a `guardianKey` or `guardianChance`) from `intelligentFindSafeEat` unless energy is at critical threshold (≤15) with no safe alternative. These are consistently poison/death sources.

**P4 — Survival mode state machine with persistent cross-turn memory**
Add persistent `botState.memory` (recentThreats, currentGoal, dangerZones) to carry information across turns within a run. Add `determineIntelligentBotMode()` that reads from this memory, not just from current-turn snapshots. Route to EMERGENCY_ESCAPE, RECOVER, WATER_PLAN, SAFE_FORAGE, EXPLORE modes. Without the cross-turn memory, the state machine is a reorganised rule list — structurally equivalent to the current approach. With it, the bot becomes goal-directed rather than purely reactive. This is the structural change; P1/P2/P3/P6 are heuristic improvements that complement it but cannot replace it.

**P5 — Fix run summary duplication (BUG-002)**
Remove `summariseCurrentBotRun("player died")` from end-of-step in both `stepRandomBot` and `stepIntelligentBot`. One line each.

**P6 — Parasite groom threshold**
Lower groom trigger to `parasites >= 1`. Two parasite deaths in latest run suggest current threshold (2) is too late.

**P7 — Death cause aggregation improvement**
Update `botDeathCauseFromRun` to prefer specific `context.source` values over `encounterAtDeath.name`.

**P8 — Terminal last-5-action trace in compact logs**
Add final 5 actions (id, layer, fitness, energy, hydration, threat present) to run summaries in compact reports. Makes post-mortem analysis tractable without needing the full log.

**P9 — Code cleanup (CODE-001)**
Fix missing `}` in `chooseIntelligentBotAction`.
