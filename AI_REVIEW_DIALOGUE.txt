--- Issue A: ecoCycleEncounterModifier() mutates state and calls addLog ---

Observation:
  ecoCycleEncounterModifier() (~line 4040) is named and used as a pure
  multiplier function — it is called in spawnNearbyEntities() and
  rollImmediateEncounter() to get a numeric modifier. However, it also
  mutates cycle.announced and calls addLog() as a side effect. This means
  the function does two unrelated things: compute a number, and announce
  a game event. A reader of the call sites sees only "get modifier" and
  has no reason to expect state mutation or logging. This is a hidden
  side effect in a function whose signature gives no indication of it.

  Additional concern: ecoCycleEncounterModifier() is called from
  spawnNearbyEntities(), which iterates over many tiles and calls it for
  each entity template at each tile. The announcement side effect fires
  on the first call that passes the !cycle.announced check, but since the
  loop continues after that, subsequent iterations call the function again
  (cycle.announced is now true, so addLog doesn't fire again). This is
  correct by accident — it works because announced is flipped on the first
  call — but the structure is fragile: if call order ever changes, or if
  spawnNearbyEntities() is ever called before rollImmediateEncounter() on
  the same turn, the announcement timing shifts silently.

Proposed change:
  Move the announcement logic into updateEcoCycleState(), at the point where
  cycle.active is first set (lines ~4029-4034). Remove cycle.announced, the
  announcement block from ecoCycleEncounterModifier(), and the ensureEcoCycle
  call from ecoCycleEncounterModifier() if it is no longer needed there.
  ecoCycleEncounterModifier() becomes a true read-only function.

Justification:
  Single responsibility. The modifier function should only compute and return
  a value. Side effects belong in the update function that manages state.
  Static analysis only.

Risk/Tradeoff:
  Low. The announcement moves from "first encounter roll using the modifier"
  to "the turn the cycle starts." The log message content is unchanged.
  No gameplay logic changes. cycle.announced can be removed, slightly
  simplifying the state object.

Validation plan:
  node --check on extracted JS. Playtest: confirm cue appears the turn a
  cycle starts. Bot 200+ turns: no duplicate announcements, no state errors,
  ecoCycle present in debug report.

--- Issue B: isPatch5SmallDampFauna() reused as insect detector in Patch 8 ---

Observation:
  ecoCycleEncounterModifier() calls isPatch5SmallDampFauna(template, key)
  (line ~4048, comment: "reuse Patch 5 helper") to identify insects for the
  insectBloom cycle modifier. The function's name encodes its patch origin,
  not its purpose. Its logic (size <= 4, not predatory, or name matches an
  insect/small-fauna regex) is broader than "insect" — it also matches frogs,
  crabs, snails, slugs. If Patch 5's definition of small damp fauna ever
  changes, Patch 8's insect bloom modifier changes with it silently.

Proposed change:
  Two options:
  A) Rename isPatch5SmallDampFauna to a purpose-describing name such as
     isSmallFauna() or isSmallNonPredator(). This fixes the naming and
     makes the reuse intentional and visible rather than incidental.
  B) Leave the function as-is for Patch 5, and add a focused
     isInsectTemplate(template, key) function for Patch 8 that only
     matches actual insects. This is stricter but means the insectBloom
     cycle only fires for insects, not frogs and crabs.

  Lean: Option A is safer (zero behaviour change, pure rename). Option B
  is more semantically correct for the insectBloom cycle but is a behaviour
  change and out of scope for a code quality patch.

Justification:
  Naming clarity and decoupling of patches. Static analysis only.

Risk/Tradeoff:
  Option A: rename only, no behaviour change. Low risk. Requires updating all
  three call sites (ecoCycleEncounterModifier, weatherEcologyEncounterModifier,
  and the function definition itself).
  Option B: behaviour change — out of scope here.

Validation plan:
  Option A: node --check. Grep confirms all call sites updated. Bot run
  to confirm no change in encounter distribution.

--- Issue C: weatherMod ternary duplicated across two functions ---

Observation:
  The expression:
    environment.weather === "heavy rain" ? 0.65 : environment.weather === "mist" ? 0.85 : 1
  appears verbatim at line ~5288 (spawnNearbyEntities) and line ~8075
  (rollImmediateEncounter). Both also use the same magic numbers (0.65, 0.85).
  If the weather modifier values need to change, they must be changed in two
  places. There is already a weatherEcologyEncounterModifier() function for
  more complex weather logic; this inline expression appears to be a simpler
  earlier version that was never consolidated.

Proposed change:
  Extract into a small utility function, e.g.:
    function baseWeatherSpawnModifier() {
      if (environment.weather === "heavy rain") return 0.65;
      if (environment.weather === "mist") return 0.85;
      return 1;
    }
  Replace both inline ternaries with a call to this function.

Justification:
  DRY. Single source of truth for these values. Static analysis only.

Risk/Tradeoff:
  Very low. Pure refactor — no behaviour change. The new function is three
  lines and has no dependencies.

Validation plan:
  node --check. Bot run confirming encounter rates are unchanged.
